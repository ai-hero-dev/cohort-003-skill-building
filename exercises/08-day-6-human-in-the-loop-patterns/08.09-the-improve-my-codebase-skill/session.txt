  1. Quiz Subsystem вҖ” Merge scoring, CRUD, and XP awarding into one deep module

  - Cluster: quizService.ts, quizScoringService.ts, xpService.ts, plus quiz UI logic in the lesson route
  - Why they're coupled: Quiz scoring imports raw better-sqlite3 alongside Drizzle ORM (mixed abstraction

  management, and attempt recording all share the same DB tables and must be coordinated.
  - Dependency category: Local-substitutable (SQLite via Drizzle, testable with in-memory DB)
  - Test impact: quizXp.test.ts, quizScoringService.test.ts would be replaced by boundary tests on a unified
   quiz module

  ---
  2. Student Progress System вҖ” Consolidate the 6-file progress calculation sprawl

  - Cluster: progressService.ts, courseService.ts, lessonService.ts, moduleService.ts,
  videoTrackingService.ts, streakService.ts
  - Why they're coupled: Understanding course progress requires bouncing across 6 service files. The
  dashboard makes 4+ redundant DB round-trips per course (N+1 pattern). calculateProgress() internally
  re-queries lesson/module data that callers already have.
  - Dependency category: Local-substitutable (SQLite/Drizzle)
  - Test impact: moduleCompletion.test.ts and scattered progress-related assertions would be replaced by a
  single progress module's boundary tests

  ---
  3. Enrollment + Side Effects вҖ” Wrap enrollment, notification, and coupon redemption into a transactional
  domain operation

  - Cluster: enrollmentService.ts, notificationService.ts, couponService.ts, purchaseService.ts
  - Why they're coupled: enrollUser() calls createNotification() as a side effect with no transaction safety
   вҖ” enrollment succeeds but notification can fail silently. Coupon redemption similarly coordinates team
  lookups, enrollment, and multi-recipient notifications in one non-atomic flow.
  - Dependency category: Local-substitutable (SQLite/Drizzle, all in-process side effects)
  - Test impact: Individual service tests would be replaced by enrollment-flow boundary tests that verify
  the complete outcome (enrollment + notification + XP)

  ---
  4. Monolithic Route Files вҖ” Extract instructor course management into a domain module

  - Cluster: instructor.$courseId.tsx (1,886 LOC), courses.$slug.lessons.$lessonId.tsx (1,400+ LOC)
  - Why they're coupled: These routes mix HTTP handling, form validation, business logic orchestration, and
  React state management. The instructor route handles module CRUD, lesson CRUD, drag-and-drop reordering,
  publishing вҖ” all inline. Understanding any single feature requires reading the entire file.
  - Dependency category: In-process (pure orchestration logic that could be extracted)
  - Test impact: Currently untestable without full route integration tests; extracting would enable direct
  unit testing of course management logic

  ---
  5. Analytics Aggregation вҖ” Deepen the analytics service to hide query complexity

  - Cluster: analyticsService.ts (426 LOC), plus schema dependencies on purchases, enrollments,

  - Dependency category: Local-substitutable (SQLite/Drizzle)
  - Test impact: Would enable comprehensive boundary tests for analytics calculations, replacing fragile
  date-logic testing

в—Ҹ Three Interface Designs

  Design A: Minimal (1-3 entry points)                                                                      
                                                                                                            
  // 3 entry points: read, submit, save                                                                     
  getQuizForLesson(lessonId, userId) вҶ’ { quiz, bestAttempt, latestAttempt }                                 
  submitQuiz(userId, quizId, answers)  вҶ’ SubmitResult (includes xpAwarded flag)                             
  saveQuiz(input, lessonId)            вҶ’ void (delete+recreate for instructor)                              
                                                                                                            
  - Lesson route goes from 5 imports across 3 files вҶ’ 1 import, 2 calls                                     
  - Instructor CRUD wizard keeps importing granular functions from existing quizService.ts                  
  - getQuizStats/getUserQuizHistory left out вҖ” would need a separate analytics module                       
  - Simplest possible interface, but instructor authoring lives outside the boundary

  Design B: Flexible (factory + scoring strategy)                                                           
                                                                                                            
  const quiz = createQuizModule({ db?, scoringStrategy?, quizPassXp? });

  quiz.getBestAttempt(userId, quizId)     вҶ’ AttemptRecord | null

  quiz.getStats(quizId)                   вҶ’ QuizStats                                                       
  quiz.submitAttempt({ userId, quizId, answers }) вҶ’ AttemptResult

  My Recommendation: Design B                                                                               
                                                                                                            
  Design B is the strongest choice for this codebase. Here's why:
                                                                                                            
  1. It hits the sweet spot. Design A is too minimal вҖ” it leaves instructor CRUD outside the boundary, which
   means quizService.ts still exists as a shallow module alongside the deep one. Design C is too ambitious вҖ”
   the namespaced sub-objects add complexity for the instructor route that currently works fine with flat   
  function imports.
  2. The factory pattern is the key insight. createQuizModule({ db }) structurally prevents the raw-SQL
  drift that caused the better-sqlite3 problem. Tests inject an in-memory DB directly вҖ” no vi.mock("~/db")  
  hacks needed. This is the single biggest testability improvement.
  3. It absorbs stats and history. Designs A omits getQuizStats/getUserQuizHistory, leaving them orphaned.  
  Design B rewrites them in Drizzle internally and exposes them cleanly, eliminating the raw SQL entirely.  
  4. saveQuiz replaces the instructor CRUD orchestration. The instructor quiz route currently does
  deleteQuiz вҶ’ createQuiz вҶ’ loop(createQuestion) вҶ’ loop(createOption) вҖ” 4+ sequential calls with no         
  transaction. saveQuiz wraps this in a single db.transaction(). The granular question/option CRUD from
  quizService.ts can be deleted.                                                                            
  5. The ScoringStrategy hook is zero-cost insurance. The default covers multiple_choice and true_false. If
  question types never expand, it's just a default parameter that nobody overrides. If they do expand, the  
  extension point is already there.
                                                                                                            
  I'd take Design B's factory + injection, but borrow one thing from Design A: the xpAwarded: boolean return
   (simpler than xpAwarded: number | null). The XP amount (5) is an implementation detail the caller doesn't
   need.
